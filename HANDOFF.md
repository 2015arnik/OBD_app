# HANDOFF — как продолжить проект (для второго участника / нового чата Claude)

Этот файл нужен, чтобы работу можно было продолжить с любого аккаунта Claude без
потери контекста (например, если у одного участника кончился лимит).

## 0. Как этим пользоваться (второй участник)
1. Установите **Claude Desktop**, откройте режим **Cowork**.
2. Склонируйте репозиторий и подключите папку к Cowork:
   `git clone https://github.com/2015arnik/OBD_app.git`
   затем в Cowork подключите папку `OBD_app` (кнопка доступа к папке).
3. Откройте и прочитайте: **этот HANDOFF.md** и **docs/СТРАТЕГИЯ.md**.
4. Вставьте новому чату Claude «стартовый промпт» из раздела 9 — и продолжайте.

## 1. Что за проект
Хакатон. Система организации поздравлений с днём рождения. Цель — **коэффициент 1.4**
(максимум): полное ядро + все усложнения + две платформы (веб и мобильное).
Полное ТЗ, карта баллов, архитектура и план — в `docs/СТРАТЕГИЯ.md`.

## 2. Стек и ключевые решения (не менять без причины)
- **Бэкенд:** Java 17 + Spring Boot 3.3.5, сборка Maven (`mvn`).
- **ВАЖНО: без Lombok.** Lombok не завёлся в Maven, поэтому геттеры/сеттеры пишем
  руками. Не добавляйте Lombok обратно.
- БД: **H2 in-memory** (данные сбрасываются при рестарте — наполняем seed'ом).
  ORM: Spring Data JPA. В проде меняется на PostgreSQL правкой `application.properties`.
- Авторизация: **JWT (jjwt 0.12.6) + BCrypt**. Токен — в заголовке `Authorization: Bearer <token>`.
- Документация: **springdoc**, Swagger UI на `/swagger-ui.html`.
- **Веб:** React + Vite + PWA (ещё не начато).
- **Мобильное (iOS/Android):** React Native + Expo (ещё не начато).

## 3. Текущее состояние
Готово и проверено (`mvn spring-boot:run` поднимается, `/health` = `{"status":"ok"}`):
- Модель данных: 9 сущностей + 9 репозиториев (`model/`, `repository/`).
- Авторизация: `POST /auth/register`, `POST /auth/login`, `GET /auth/me`, `GET /health`.
- CORS, JWT-фильтр, резолвер `@CurrentUser`.

Ещё НЕ сделано: API ядра, реалтайм-чат, усложнения (сбор/мок-банк, планировщик,
календарь, админка), seed-данные, веб, мобильное, обучающий гайд по коду.

## 4. Архитектура и соглашения (соблюдать!)
- Пакеты: `com.obd.model` (сущности), `repository` (JPA), `dto` (запросы/ответы),
  `web` (контроллеры), `security`, `config`. Добавлять по мере роста: `service`
  (бизнес-логика), `ws` (вебсокет-чат), `scheduler` (напоминания).
- Сущности «плоские»: внешние ключи — простые `Long` поля (`ownerId`, `targetUserId`
  и т.п.), без JPA-связей `@ManyToOne/@OneToMany`. Членство в группе — отдельная
  сущность `GroupMembership(userId, groupId)`. Это сделано осознанно: меньше «магии»,
  проще защищать код.
- Текущий пользователь: контроллер объявляет параметр `@CurrentUser User me`,
  резолвер подставит залогиненного или вернёт 401. Пример — `AuthController.me`.
- Ошибки: `throw new ResponseStatusException(HttpStatus.XXX, "сообщение")`.
- DTO: простые классы с `public`-полями + валидация `@NotBlank/@Email` на входных.
  В ответах часто возвращаем сущность напрямую (пароль скрыт `@JsonIgnore`).
- Геттеры/сеттеры — вручную (без Lombok).

## 5. Рабочий процесс (репозиторий + Cowork) — критично
- Репозиторий: https://github.com/2015arnik/OBD_app, ветка `main`.
- Claude **пишет файлы прямо в подключённую папку** `~/Desktop/OBD_app`.
  Человек делает `git add/commit/push` сам на Маке.
- Песочница Claude в этой папке умеет **создавать/перезаписывать**, но **НЕ удалять**
  файлы, и **не должна запускать git** (оставляет `.git/index.lock`). Значит:
  - Claude — только создаёт/меняет файлы.
  - git и удаление файлов — только человек в терминале Мака.
- В командных блоках **НЕ писать комментарии `# ...`** — zsh ругается
  (`unknown file attribute`), особенно если в комментарии есть скобки `()`.

## 6. Как запускать и проверять
```
cd ~/Desktop/OBD_app/backend
mvn spring-boot:run
```
Нужен JDK 17. Проверка: `http://localhost:8080/health`, Swagger `http://localhost:8080/swagger-ui.html`,
консоль БД `http://localhost:8080/h2-console` (JDBC URL `jdbc:h2:mem:obd`, user `sa`).
После каждого этапа: собрать → проверить эндпоинты в Swagger → `git add/commit/push`.

## 7. План дальше (по приоритету к 1.4)

### Этап A — API ядра  ← СЛЕДУЮЩИЙ ШАГ
Контроллеры + при необходимости сервисный слой:
- `UserController`: `GET /users` (все пользователи, сортировка по ближайшему ДР),
  `GET /users/{id}` (карточка: профиль + группы + вишлист), `PATCH /users/{id}` (свой профиль).
- `GroupController`: `GET /groups`, `POST /groups` (creatorId = текущий), `POST /groups/{id}/join`,
  `GET /groups/{id}/members`.
- `GiftController`: `GET /users/{id}/gifts`, `POST /gifts` (ownerId = текущий),
  `PATCH /gifts/{id}` (статус: резерв/куплено), `DELETE /gifts/{id}` (только владелец).
- `SubscriptionController`: `POST /subscriptions` (user или group), `DELETE /subscriptions/{id}`,
  `GET /subscriptions` (мои).
- `NotificationController`: `GET /notifications` (мои), `POST /notifications/{id}/read`.
Критерий готовности: все эндпоинты вызываются в Swagger, данные сохраняются в H2.

### Этап B — Реалтайм-чат (WebSocket)
- WS `/ws/chat/{userId}`: «комнаты» по targetUserId, новое сообщение → сохранить
  `ChatMessage` → разослать всем в комнате.
- **Приватность:** если текущий пользователь == `userId` — запретить (именинник не видит
  свой чат). То же для `GET /users/{id}/chat` (история).
Критерий: в двух вкладках сообщения приходят мгновенно; именинник не видит.

### Этап C — Усложнения
- **Сбор средств + мок-банк:** `POST /mock-bank/charge` (возвращает фейковый `txnId` и
  «успех»), `POST /fundraisers`, `POST /fundraisers/{id}/contribute`, `GET /fundraisers/{id}`
  (прогресс: goalAmount/collectedAmount).
- **Планировщик `@Scheduled`:** раз в сутки — уведомления подписчикам о ДР через N дней;
  за M дней до ДР — автосоздание сбора + уведомление со ссылкой.
- **Календарь:** `GET /users/{id}/calendar.ics` (VEVENT, ежегодный повтор) + ссылка «добавить
  в Google Календарь».
- **Админка:** эндпоинты `/admin/*` (только `admin`), импорт пользователей/групп из CSV/JSON.

### Этап D — Seed-данные
`CommandLineRunner`: несколько пользователей с разными ДР (в т.ч. ближайшие на днях),
группы («972501 ТГУ», «сборная по волейболу»), подарки, подписки. Пароль у всех
одинаковый (например `password`) — указать в README.

### Этап E — Веб (React + Vite + PWA)
Экраны: Логин/Регистрация, Люди (список с ДР), Карточка друга (профиль, группы, вишлист,
чат), Мой вишлист, Группы, Уведомления, Сбор. Axios-клиент с токеном, WebSocket для чата,
`vite-plugin-pwa` (manifest + service worker).

### Этап F — Мобильное (React Native + Expo)
Те же экраны и тот же API. Запуск на Маке: iOS-симулятор или Expo Go на айфоне.

### Этап G — Обучающий гайд по коду + репетиция защиты
Построчный разбор ключевых файлов простым языком (для защиты). Прогон демо-сценария
(см. `docs/СТРАТЕГИЯ.md`, разделы 10–11).

## 8. Известные грабли (чтобы не повторять)
- Lombok в Maven не сработал → убрали, геттеры/сеттеры руками.
- Песочница Claude: create/overwrite — да, delete — нет; git из песочницы не запускать.
- zsh: без `#`-комментариев в командах (скобки в комментах ломают строку).
- H2 in-memory сбрасывается при рестарте → наполнять через seed (Этап D).
- Таблица группы называется `groups` (GROUP — зарезервированное слово).
- Именование столбцов camelCase → snake_case (`userId` → `user_id`); в `@UniqueConstraint`
  указывать snake_case.
